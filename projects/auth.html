Why Build Auth From Scratch?

Most frameworks handle auth for you. Laravel has it built-in. But I wanted to understand what's actually happening. How does password_hash() work? Why do we use password_verify() instead of comparing strings? What makes sessions secure?

Building this from scratch taught me more about security than any tutorial. I made mistakes (stored plain text passwords once ‚Äî never again), learned from them, and now I actually understand what Laravel's Auth facade does under the hood.

Security Features

Password Hashing

Bcrypt hashing with password_hash(). Passwords are never stored in plain text.

SQL Injection Prevention

100% prepared statements. Every single database query uses parameter binding.

Session Security

Regenerated session IDs on login. HTTP-only cookies. Session timeout after inactivity.

Password Reset

Token-based reset with expiration. Tokens are single-use and time-limited.

Security Note: This is a demo system. In production, you'd add: HTTPS enforcement, rate limiting on login attempts, email verification, CSRF tokens on all forms, and more comprehensive logging.

The Password Hash That Changed Everything

My first version stored passwords like this: md5($password). Then I learned about rainbow tables. Then I learned about bcrypt. Now I do it right:

// Registration: Hash the password $hashed = password_hash($password, PASSWORD_BCRYPT); // $hashed looks like: $2y$10$eImiTXuWVxfM37uY4JANjQ... // Login: Verify against stored hash if (password_verify($entered_password, $stored_hash)) { // Regenerate session to prevent fixation session_regenerate_id(true); $_SESSION['user_id'] = $user['id']; $_SESSION['logged_in'] = true; }

Password Reset Flow

This was surprisingly complex to get right. The token needs to be secure, expire after use, and be tied to the specific user:

  1. User enters email address
  2. Generate a cryptographically secure token: bin2hex(random_bytes(32))
  3. Store hashed token in database with expiration (1 hour)
  4. Send email with reset link containing the token
  5. User clicks link ‚Üí verify token exists and hasn't expired
  6. User sets new password ‚Üí delete the used token

Lessons I Learned the Hard Way

üîê Never roll your own crypto

I thought about implementing my own hashing. Then I read about all the ways that can go wrong. Use password_hash(). Always.

üîê Session IDs must be regenerated

Session fixation attacks are real. Regenerate the session ID whenever privilege level changes (login, logout, admin access).

üîê Timing attacks are a thing

password_verify() is timing-safe. String comparison (===) is not. Small details like this matter for security.

"Security isn't a feature you add at the end. It's a mindset you have from line one."
